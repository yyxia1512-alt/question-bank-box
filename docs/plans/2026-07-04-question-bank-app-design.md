# 鸿蒙离线题库练习 App 设计文档

## 1. 背景与目标

本项目要开发一个运行于鸿蒙 6 的离线题库练习 App。初版题库来自《专业知识考试题库（2026年修订版）.pdf》，PDF 仅作为开发期数据源，不进入用户侧解析流程。预处理后产出结构化种子题库，App 首次启动时导入本地数据库。完整题库数据不上传到 public 仓库，只通过本地或私有发布渠道单独集成。

App 的定位不是单纯刷题器，而是“离线题库练习 + 本地题库维护”的生产型单机工具。普通用户可以练习题库，也可以本地新增、编辑、删除题目，修改答案和解析。用户修改内容前需要二次确认，内容正确性由用户承担；软件必须保证修改后的本地数据、错题状态、练习进度和统计状态一致。

## 2. 已确认范围

### 2.1 功能范围

- 全量刷题：直接使用当前启用题库的全部题目。
- 模块刷题：先选择模块；不选择题型时刷该模块全部题目。
- 模块内题型刷题：在模块下选择单选、多选或判断后，只刷该模块下对应题型。
- 继续上次练习：恢复未完成的练习会话。
- 错题集：由答题记录和当前题目版本推导，支持手动移除展示。
- 题库维护：支持新增题目、编辑题干、选项、答案、解析、模块和题型，支持软删除。
- 固定格式导入导出：用户新增或迁移题库只能使用结构化文件，不支持用户上传 PDF 解析。
- 本地统计：展示总进度、模块正确率、错题数和最近练习情况。

### 2.2 非目标

- 不做联网同步。
- 不做账号系统。
- 不做云端题库。
- 不做用户侧 PDF 导入或 PDF 解析。
- 不做考试监考、防作弊和多人协作。
- 初版不做跨模块按题型筛选入口，避免筛选模型过早复杂化。

## 3. 产品流程设计

首页采用底部导航或侧边导航，核心入口为“练习”“错题”“题库”“统计/设置”。

“练习”是默认首页，提供三个高频入口：全量刷题、模块刷题、继续上次练习。模块刷题进入模块列表，每个模块显示题量、已练数量和正确率。进入某个模块后，用户可以选择“全部题型、单选、多选、判断”。开始练习前创建 `practice_session`，记录筛选条件、题目顺序、当前位置和练习模式。

练习页围绕答题效率设计：展示题干、选项、提交按钮、上一题、下一题、收藏/标记和解析。单选和判断提交后即时判定；多选必须用户主动提交，避免未选完就误判。答错后自动进入错题候选。错题页允许用户手动移除展示，但不删除历史错误记录。

“题库”负责维护：模块管理、题目列表、题目详情编辑、新增题目、软删除、导入导出。凡是影响答案、题干、选项或题型的保存都需要二次确认，并触发本地一致性刷新。“统计/设置”只展示本机数据，不依赖网络。

## 4. 技术方案

### 4.1 技术栈

采用 HarmonyOS 6 原生路线：

- ArkTS
- ArkUI
- Stage 模型
- 本地关系型数据库保存结构化题库、答题记录和统计
- Preferences 保存轻量设置和最近使用偏好

当前未发现值得直接 fork 的成熟鸿蒙离线题库开源项目。工程应自建，参考 HarmonyOS Samples 的工程组织和 ArkUI 写法即可。

### 4.2 工程分层

建议目录按职责分层：

- `pages`：页面组合和导航。
- `features`：练习、错题、题库管理、统计设置等业务用例。
- `domain`：题型、答案判定、筛选、错题规则、导入校验等纯逻辑。
- `data`：关系型数据库、Preferences、种子题库导入、事务和 repository。
- `common`：通用 UI 组件、错误码、工具函数。

页面层不能直接拼 SQL，也不能直接做答案判定。答案判定统一放在 `AnswerEvaluator`。练习题目列表由 `PracticeSessionService` 生成，支持全量、模块、模块加题型三种筛选。题库修改统一通过 `QuestionAdminService`，这是唯一能改题库的业务入口。

## 5. 数据模型设计

核心表建议如下：

- `modules`：模块信息。
- `questions`：当前有效题目，包含模块、题型、题干、解析、状态、版本号、创建来源。
- `options`：题目选项。判断题内部可统一成“正确、错误”两个选项。
- `question_revisions`：题目每次编辑前后的快照。
- `practice_sessions`：练习会话。
- `answer_records`：每次答题事实记录。
- `question_stats`：题目维度的派生统计。
- `module_stats`：模块维度的派生统计。
- `wrong_question_overrides`：用户手动移除错题展示等覆盖状态。
- `import_exports`：导入导出记录。

`answer_records` 是事实表，每次答题记录题目 ID、当时题目版本、用户答案、当时正确答案、是否正确、耗时和所属练习会话。历史记录绝不回写。`question_stats` 和 `module_stats` 是派生表，可以从事实表重建。

用户修改题干、选项、答案、解析或题型后，题目 `revision` 自增，旧历史记录保留为旧版本记录，当前掌握状态重置为“需重新练习”。不按新答案重新判定旧答题记录，避免制造假正确或假错误。

## 6. 一致性刷新设计

题库修改必须在本地事务中完成。标准流程：

1. 校验输入。
2. 写入 `question_revisions`。
3. 更新 `questions` 和 `options`。
4. 标记受影响统计失效。
5. 重建受影响题目的 `question_stats`。
6. 刷新模块聚合统计。
7. 提交事务。
8. 通知 UI 重新加载。

导入题库这种批量操作先进入临时表或临时对象校验，全部通过后再进入数据库事务。失败时不能落半截数据。修改完成前 UI 显示刷新中；刷新失败则整笔事务回滚。这里的“后台刷新”是 App 进程内的一致性任务，不承诺应用被系统杀掉后继续运行。

删除题目采用软删除。软删除题目不进入新的练习会话，但历史答题记录仍可保留和查看。错题集不是一份独立题库，而是根据当前题目状态、答题事实和用户手动覆盖状态计算展示，避免数据漂移。

## 7. PDF 预处理设计

PDF 预处理是仓库内开发工具，不进入鸿蒙 App。当前 PDF 实测信息：

- PDF 页数：167 页。
- 总题数：475。
- 第一部分“相关法律法规”：单选 81、多选 39、判断 44。
- 第二部分“不动产登记”：单选 81、多选 65、判断 58。
- 第三部分“地籍调查”：单选 9、多选 8、判断 1。
- 第四部分“登记资料管理”：单选 10、多选 20、判断 11。
- 第五部分“综合案例”：单选 29、多选 17、判断 2。

解析策略必须使用状态机：

1. 识别模块标题。
2. 识别题型标题。
3. 按期望题号递增切题。
4. 容忍 `答案.D`、`答 案：A`、题号漏点号、题号拼接到上一题解析等异常格式。
5. 避免把案例正文中的数字误判为题号，例如年龄、年份、月份。

预处理输出必须做硬校验：总数 475、每模块每题型计数完全匹配、每题有合法答案、选项与答案匹配、无重复 ID、无孤立模块引用。解析标签缺失但内容存在的题目进入人工校验报告。

## 8. 固定导入导出格式

用户侧新增或迁移题库只支持固定结构化格式。建议使用 `qbank.zip`。首版种子题库也使用同一格式，但作为本地或私有集成产物，不提交到 public 仓库：

- `manifest.json`：题库名、版本、创建时间、格式版本、题量摘要。
- `modules.json`：模块列表。
- `questions.json`：题目、题型、选项、答案、解析、模块 ID、状态。
- `checksums.json`：各文件校验值。

导入前必须校验：

- JSON schema 合法。
- ID 唯一。
- 模块引用存在。
- 题型只允许 `single_choice`、`multiple_choice`、`true_false`。
- 单选答案只能有一个。
- 多选答案至少有一个。
- 判断答案只能是 `true` 或 `false`。
- 答案必须能匹配选项。

导入模式初版只保留两类：

- 追加导入：只新增不冲突题目。
- 覆盖导入：允许更新同 ID 题目，需要二次确认，并触发版本变更和一致性刷新。

不做用户侧 PDF 解析，不做复杂三方合并。

## 9. 关键领域服务

### 9.1 AnswerEvaluator

统一处理单选、多选和判断。输入题目快照和用户答案，输出是否正确、错误原因、缺失选项、多余选项和解析展示状态。多选答案比较不依赖用户选择顺序。

### 9.2 PracticeSessionService

负责创建和恢复练习会话。支持全量题库、模块题库、模块内题型题库三种筛选。生成题目列表时排除软删除题目，并记录题目版本，避免练习过程中题库修改导致会话语义变化。

### 9.3 WrongQuestionService

负责错题展示规则。答错进入错题候选；用户可手动移除展示；题目被修改后当前状态重置为需重新练习。历史错误事实不删除。

### 9.4 QuestionAdminService

唯一的题库修改入口。负责新增、编辑、软删除、导入覆盖、revision 快照、二次确认后的事务保存和一致性刷新。

### 9.5 ImportExportService

负责 `qbank.zip` 的导入导出、schema 校验、checksum 校验、临时解析和事务落库。

## 10. 首启与恢复流程

首启流程：

1. 检查数据库版本。
2. 建表或执行迁移。
3. 检查是否已导入种子题库。
4. 导入内置 `qbank.zip`。
5. 重建统计。
6. 进入首页。

如果首启导入失败，App 应给出可恢复状态，例如重新导入种子或进入空题库模式。种子题库重复导入必须幂等，不能重复生成同一批题目。

## 11. 测试与验收

### 11.1 题库数据验收

- PDF 预处理总题数为 475。
- 模块和题型计数完全匹配。
- 每题有合法答案。
- 每题答案与选项匹配。
- 无重复题目 ID。
- 无孤立模块引用。
- 人工抽检覆盖每个模块、每种题型、跨页题、案例题和异常格式题。

### 11.2 核心逻辑测试

- 单选：正确、错误、空答案、非法答案。
- 多选：正确、少选、多选、顺序不同、空答案、非法答案。
- 判断：正确、错误、空答案。
- 筛选：全量、模块、模块加题型。
- 继续练习：恢复位置和题目顺序。
- 软删除题目不进入新练习。
- 修改答案后历史保留，当前状态重置为需重新练习。

### 11.3 一致性测试

- 题目编辑事务成功后，题库、统计、错题展示一致。
- 答案修改失败时，所有相关数据回滚。
- 覆盖导入失败时，不污染现有题库。
- 批量导入成功后，模块统计和题目统计一致。
- App 重启后状态与修改前后一致。

### 11.4 鸿蒙端体验验收

- 离线启动正常。
- 页面跳转正常。
- 长题干可滚动阅读。
- 多选必须提交后判定。
- 解析展示清晰。
- 题库编辑有二次确认。
- 导入导出错误提示明确。
- 真机或模拟器上无明显卡顿、闪退和状态错乱。

## 12. 生产风险与约束

- 用户可修改首版题库，因此软件不能保证内容绝对正确，只保证数据状态一致。
- PDF 解析只在开发期使用，用户新增题库必须走固定格式。
- 单机 App 没有云端兜底，导入导出是用户备份和迁移的主要手段。
- 本地事务和统计重建是底线，不能为了开发速度把错题和统计做成容易漂移的独立状态。
- 初版应优先保证题库数据正确、练习流程稳定、修改后一致性可靠，再考虑视觉美化和高级练习算法。

## 13. 实施顺序建议

1. 建立 PDF 预处理工具和种子题库包，先证明 475 题可稳定结构化。
2. 搭建 HarmonyOS 6 ArkTS 工程骨架。
3. 实现数据库 schema、迁移和种子导入。
4. 实现 `AnswerEvaluator`、`PracticeSessionService` 和基础练习页面。
5. 实现答题记录、统计和错题展示。
6. 实现题库维护、revision、二次确认和一致性刷新。
7. 实现固定格式导入导出。
8. 完成自动化测试、数据校验报告和真机/模拟器验收。
